>[!Question] Q1:把造型函数(shaping funtion)整合进来，来让HSB和RGB的转换中强调某些特定值并且弱化其他的。

回顾一下当前这段色轮代码里，三个有意义的量分别是：

```glsl
color = hsb2rgb(vec3((angle/TWO_PI)+0.5, radius, 1.0));
//                          ↑Hue           ↑Sat    ↑Bright
```

如果直接对算出来的 `color.r/g/b` 做 pow/sqrt，相当于在**RGB空间**里硬调，效果会比较生硬、不直观，因为 RGB 三个通道之间互相牵连，调一个通道容易把整体颜色"带偏"。

**更合理、的做法，是在 HSB 空间里，对 Hue/Saturation/Brightness 这几个语义清晰的量分别下手**——因为它们三个互相独立，改一个不会牵连另外两个，效果更可控、更容易看出"强调/弱化"的意图。

这其实是整本 Book of Shaders 这一章的核心思路：**造型函数本质是"输入一个0~1的值，输出另一个0~1的值，但强调/压缩某些区间"**，而 HSB 的三个分量正好都是天然的 0~1 数值，非常适合套造型函数。

---
### 一：用 pow 强调中心的纯色/边缘的纯色（操作 radius，即饱和度）

```glsl
float radius = length(toCenter) * 2.0;
radius = pow(radius, 3.0);   // 用pow强调"边缘"，压缩"中心附近"
color = hsb2rgb(vec3((angle/TWO_PI)+0.5, radius, 1.0));
```

效果：之前我们讲过 `pow(x,3.0)` 是"先慢后快"曲线，套在饱和度上意味着：**靠近中心的大片区域饱和度都被压得很低（更长时间停留在"发白"状态），只有接近最外圈才迅速冲到饱和度=1**。视觉上，白色的"核心"区域会变得更大、更明显，鲜艳的色环被挤到更靠外的薄薄一圈。

反过来，用 sqrt：

```glsl
radius = sqrt(radius);   // 先快后慢
```

效果相反：**饱和度很快就冲高，画面中心附近很快就变得鲜艳，只有极小的中心点是白色**，白色核心被压缩得很小，大部分区域看起来都很饱和。

---

### 玩法二：强调/弱化某些色相区间（操作 angle算出来的 Hue）

```glsl
float hue = (angle/TWO_PI) + 0.5;
hue = pow(hue, 2.0);   // 让某些颜色"挤在一起"，某些颜色"被拉宽"
color = hsb2rgb(vec3(hue, radius, 1.0));
```

效果：色相本身是环形的 0~1，套 pow 之后，**色相变化的"速度"不再均匀**——原本360°平均分布的颜色，会变成某一段颜色（比如红到黄）占据更大的角度范围，显得这段色彩"被强调/拉长"了，而另一段颜色（比如蓝到紫）会被压缩到很窄的角度范围里，显得"被弱化/挤扁"了。

---

### 玩法三：强调明度的层次感（操作 Brightness）

当前代码里明度固定是 `1.0`，没什么可调的，但你可以让明度也参与造型，比如结合 radius：

```glsl
float brightness = 1.0 - pow(radius, 4.0) * 0.3;  // 边缘稍微变暗，强调中心的亮
color = hsb2rgb(vec3(hue, radius, brightness));
```

这样可以做出"边缘渐暗的晕影（vignette）效果"，强调画面中心的视觉重量。

这个练习问的是个真实存在的色彩理论问题，先搞清楚问题本质，再想怎么用 shader 解决。
## 问题出在哪：RGB 色环 vs RYB 色环

我们代码里用的是标准的 **HSB/RGB 色相环**，它的色相分布是数学/物理上"均匀"的（基于光的加色混合原理），互补色关系是：

```
红(0°) 对面是 青(180°)
绿(120°) 对面是 品红(300°)
蓝(240°) 对面是 黄(60°)
```

但**传统美术、色彩理论课、绘画里教的色环（RYB，Red-Yellow-Blue）**，是基于颜料的减色混合习惯，互补色关系是大众更熟悉的：

```
红 对面是 绿
黄 对面是 紫
蓝 对面是 橙
```

这就是题目说的："红色对面应该是绿色，但我们这里是青色"——**本质是两套不同的色彩理论体系，色相环上颜色排列的角度对应关系不一样**。

---

## 解决思路：不是简单平移角度，而是要"扭曲"角度的分布

第一直觉可能是"把 angle 加一个偏移量，把红和绿调到对面"——但**这样做只能整体旋转色环，不能改变"谁和谁相对"的关系**。比如简单加偏移：

```glsl
hue = (angle/TWO_PI) + 0.5 + 0.33;  // 只是整体转了一个角度
```

这样所有颜色都跟着转了，红色现在可能转到了原来青色的位置没错，但**绿色也同时被转到了别的位置，红绿关系并没有被"重新配对"**，本质问题没解决。

真正需要做的，是把"角度→色相"这个映射关系，从"线性对应"换成"非线性对应"**——也就是说，需要一个**重映射函数**：给定一个角度，输出一个"经过调整的"色相值，使得：

```
0°（红）   保持在 0°
角度走到180° 时，对应的颜色应该是"绿"，而不是青
```

而 RGB色相环里，"绿"原本是在 **120°** 的位置（不是180°）。

---

## 具体怎么实现：分段线性映射（Piecewise Remapping）

这是**造型函数**应用的一个典型实战案例——把"输入0~1，输出0~1"的映射关系，按照需求重新设计。

RYB色环和RGB色环的对应关系大致是这样（这是一个常见的近似映射表，业界有几个版本，这里给一个简化的6段对应）：

| RYB角度 (0~1) | 对应的实际RGB色相 (0~1) |
| ----------- | ---------------- |
| 0.0 (红)     | 0.0              |
| 1/6 (橙)     | 0.083            |
| 2/6 (黄)     | 0.167            |
| 3/6 (绿)     | 0.333            |
| 4/6 (蓝)     | 0.667            |
| 5/6 (紫)     | 0.833            |
| 1.0 (红)     | 1.0              |

也就是说，RYB色环里"红到黄"这一段（0~1/3），其实只对应RGB色相里很窄的一段（0~1/6）；而RYB里"黄到绿"这一段，对应RGB色相里"1/6到1/3"。这是因为人眼/颜料感知的色彩分布并不是线性均匀的，需要这样不均匀地"拉伸/压缩"来修正。

### 写成 GLSL 代码：用 if 分段处理

```glsl
float ryb2rgbHue(float t) {
    // t 是 0~1 的 RYB 色相输入
    if (t < 1.0/6.0) {
        return mix(0.0, 1.0/12.0, t / (1.0/6.0));
    } else if (t < 2.0/6.0) {
        return mix(1.0/12.0, 1.0/6.0, (t - 1.0/6.0) / (1.0/6.0));
    } else if (t < 3.0/6.0) {
        return mix(1.0/6.0, 1.0/3.0, (t - 2.0/6.0) / (1.0/6.0));
    } else if (t < 4.0/6.0) {
        return mix(1.0/3.0, 2.0/3.0, (t - 3.0/6.0) / (1.0/6.0));
    } else if (t < 5.0/6.0) {
        return mix(2.0/3.0, 5.0/6.0, (t - 4.0/6.0) / (1.0/6.0));
    } else {
        return mix(5.0/6.0, 1.0, (t - 5.0/6.0) / (1.0/6.0));
    }
}
```

然后在 main 里这样用：

```glsl
float rybHue = (angle/TWO_PI) + 0.5;     // 先算出原始角度对应的0~1值
float fixedHue = ryb2rgbHue(rybHue);     // 重映射成RYB对应的"正确"颜色
color = hsb2rgb(vec3(fixedHue, radius, 1.0));
```

---

## 这正是"造型函数"的核心思想

回到你前一个问题问的"造型函数"主题——**这个分段映射函数 `ryb2rgbHue` 本身就是一个造型函数**：

```
输入: 0~1 (RYB色相)
输出: 0~1 (修正后的RGB色相)
```

只不过它不是用单一的 `pow`/`sqrt` 公式，而是**用6段不同斜率的线性插值拼接起来**，本质上和 `pow`/`sqrt` 是同一类工具——都是"重新设计输入和输出之间的对应关系，强调某些区间、压缩其他区间"。

这里"红到黄"那一段被压得很窄（0~1/6的输出对应0~1/3的输入），是因为现实里红到黄这段在RYB色环上"视觉跨度"比较大，需要在RGB色相环上用更窄的角度去表现，才能让两套色环"看起来"是同一种排列顺序。
