```glsl
#ifdef GL_ES
precision mediump float;
#endif

uniform vec2 u_resolution;
uniform float u_time;

//  Function from Iñigo Quiles
//  https://www.shadertoy.com/view/MsS3Wc
vec3 hsb2rgb( in vec3 c ){
    vec3 rgb = clamp(abs(mod(c.x*6.0+vec3(0.0,4.0,2.0),
                             6.0)-3.0)-1.0,
                     0.0,
                     1.0 );
    rgb = rgb*rgb*(3.0-2.0*rgb);
    return c.z * mix(vec3(1.0), rgb, c.y);
}

void main(){
    vec2 st = gl_FragCoord.xy/u_resolution;
    vec3 color = vec3(0.0);

    // We map x (0.0 - 1.0) to the hue (0.0 - 1.0)
    // And the y (0.0 - 1.0) to the brightness
    color = hsb2rgb(vec3(st.x,1.0,st.y));

    gl_FragColor = vec4(color,1.0);
}
```


这段代码是关于 **HSB（HSV）色彩空间**和 RGB 互转的经典实现，外加一个用 HSB 渲染色相-亮度渐变图的应用。

## 先搞懂 HSB 是什么

RGB 是"红绿蓝混合"的描述方式，对人类直觉不友好（很难凭感觉调出"亮一点的橙色"）。[[HSL HSV|HSB]]（Hue饱和度Brightness）更符合人类对颜色的直觉描述：

|分量|含义|范围|
|---|---|---|
|H (Hue)|色相，颜色在色环上的位置（红、黄、绿、青、蓝、紫…）|0.0~1.0（对应0°~360°）|
|S (Saturation)|饱和度，颜色的"鲜艳程度"（0=灰，1=纯色）|0.0~1.0|
|B (Brightness)|明度，颜色的"亮暗程度"（0=黑，1=最亮）|0.0~1.0|

color = hsb2rgb(vec3(st.x, 1.0, st.y))这一行的意图就是：**横轴(x)控制色相变化（红→黄→绿→青→蓝→紫），纵轴(y)控制亮度**，画出一张色相条渐变图。

## hsb2rgb 函数（重点，因为 main 实际用的是这个）

```glsl
vec3 hsb2rgb( in vec3 c ){
    vec3 rgb = clamp(abs(mod(c.x*6.0+vec3(0.0,4.0,2.0), 6.0)-3.0)-1.0, 0.0, 1.0 );
    rgb = rgb*rgb*(3.0-2.0*rgb);
    return c.z * mix(vec3(1.0), rgb, c.y);
}
```

这是 Iñigo Quilez 写的一个**无分支（branchless）**算法，用数学公式直接算出 RGB，避开了传统教科书里那种"if-else判断H落在哪个60°扇区"的写法。我们拆开看：

### 第一步：`c.x*6.0 + vec3(0.0, 4.0, 2.0)`

把色相 `c.x`（0~1）放大到 0~6 的范围（对应色环切成6个扇区），然后给 R、G、B 三个通道分别加上不同的偏移量（0, 4, 2），让三个通道在色环上错开相位——这是模拟色相环上 R/G/B 三色"互相错开120°"的关系。

### 第二步：`mod(..., 6.0)`

把结果限制在 `[0, 6)` 范围内循环（因为色相是环形的，超过6就要绕回来）。

### 第三步：`abs(... - 3.0) - 1.0`，再 clamp 到 `[0,1]`

这一串数学操作，是用绝对值函数**折叠**出一个"三角波"形状，模拟色相变化时每个通道该有的"先升后降"曲线。`clamp` 把结果限制在 `[0,1]`，超出范围的部分被削平成 0 或 1（对应色环上某些区间该通道应该是纯0或纯1的平坦区域）。

### 第四步：`rgb*rgb*(3.0-2.0*rgb)`

这是一个**smoothstep风格的多项式**，作用是把刚才那个"折线感"的三角波**磨圆滑**，让色相过渡更自然，不会有棱角。

### 第五步：`c.z * mix(vec3(1.0), rgb, c.y)`

- `mix(vec3(1.0), rgb, c.y)`：用饱和度 `c.y` 在"纯白"和"刚算出的彩色"之间插值——饱和度低就偏白，饱和度高就是纯色
- `c.z * (...)`：乘以明度 `c.z`，控制整体亮暗（0就是全黑）

## rgb2hsb 函数（反向转换，本例 main 里没用到，但定义在那里备用）

这个函数做的是相反的事：**输入RGB，算出HSB**，本例的 main 函数实际没调用它，可能是教程把"一对互逆函数"放在一起方便你对照学习/后续实验用。简单说一下思路：

```glsl
vec4 K = vec4(0.0, -1.0/3.0, 2.0/3.0, -1.0);
vec4 p = mix(vec4(c.bg, K.wz), vec4(c.gb, K.xy), step(c.b, c.g));
vec4 q = mix(vec4(p.xyw, c.r), vec4(c.r, p.yzx), step(p.x, c.r));
```

这两行用 `mix` + `step` 模拟"判断R、G、B谁最大谁最小"的逻辑（`step` 返回0或1，配合 `mix` 相当于一个无分支的条件选择），目的是找出三个通道里的最大值和最小值，这是计算色相、饱和度公式里必须知道的关键信息（HSB转换公式依赖"max-min"这个差值）。

```glsl
float d = q.x - min(q.w, q.y);
float e = 1.0e-10;
return vec3(abs(q.z + (q.w - q.y) / (6.0*d + e)), d/(q.x+e), q.x);
```

- `d` 对应 RGB 里的"最大值-最小值"（这个差值决定了饱和度和色相的计算）
- `e = 1.0e-10` 是个极小值，**纯粹是为了防止除以0**（当RGB全相等，即灰色/黑白色时，d会是0，加上e避免数学错误）
- 最终返回 `vec3(H, S, B)`

这个函数写法用"无分支数学技巧"代替了传统的 if-else，是 GPU 编程里的常见优化思路（**GPU 并行计算大量像素时，分支判断的性能开销比CPU上更明显**，所以 shader 代码里经常看到这种"用数学公式硬算，规避if判断"的写法）。

## main 函数实际做的事

```glsl
vec2 st = gl_FragCoord.xy/u_resolution;
color = hsb2rgb(vec3(st.x, 1.0, st.y));
```

- `st.x`（横向 0~1）→ 作为色相 H，所以画面**从左到右会经历红→黄→绿→青→蓝→紫→红**的完整色环
- `1.0` → 饱和度固定拉满，颜色始终鲜艳不发灰
- `st.y`（纵向 0~1）→ 作为明度 B，所以画面**从下到上由黑逐渐变亮**

最终效果是一张经典的"彩虹渐变条"：横向是色相轮转，纵向是亮度渐变，底部全黑，顶部是饱和鲜艳的纯色。

>[!question] Q
>这个`c`是什么时候定义的啊，vec3 hsb2rgb(in vec3 c)这一行吗还是一个内置的函数？以及所有vec3都能用x/y/z这三个向量来表达吗

A:两个问题都问到了关键点，分别说清楚。

## 1. `c` 是在哪里定义的

是的，就是 `vec3 hsb2rgb(in vec3 c)` 这一行——**`c` 是函数的参数名**，不是什么内置变量，纯粹是这个函数自己起的名字（取的是 color 的缩写）。

```glsl
vec3 hsb2rgb( in vec3 c ){
    //              ↑
    //   这里声明：这个函数接收一个vec3类型的参数，
    //   函数体内部，我们管它叫 c
    ...
}
```

`in` 这个关键字表示"这是一个输入参数"（GLSL 里参数可以标注 `in`、`out`、`inout`，分别表示"只读输入"、"只写输出"、"既读又写"），`in` 是最常见、默认的情况，很多时候省略不写也行：

```glsl
vec3 hsb2rgb(vec3 c) { ... }   // 不写in，效果一样，因为in是默认值
```

调用的时候，main 函数里这样传值进去：

```glsl
color = hsb2rgb(vec3(st.x, 1.0, st.y));
//               └────────┬────────┘
//              这一整个 vec3(st.x, 1.0, st.y) 
//              就是被传进去、在函数内部被称为 c 的那个值
```

也就是说，调用这一刻，`c` 就变成了 `(st.x, 1.0, st.y)` 这个具体的值，函数内部所有 `c.x`、`c.y`、`c.z` 操作的都是这三个数。

这和你之前问过的 `plot(vec2 st)` 是完全一样的机制——**函数定义时声明的参数名，只在函数体内部生效，是个"占位符"**，外部调用时传入具体值即可，不需要外部"提前定义"这个名字。

## 2. 所有 vec3 都能用 .x/.y/.z 访问吗

**能，这是 GLSL 的通用规则**，但更准确地说，GLSL 给了你**四套不同的命名习惯**，可以互换使用，本质上都是访问同样的内存位置：

|写法风格|用途场景|示例|
|---|---|---|
|`.x .y .z .w`|通用坐标/数值|`c.x`, `c.y`, `c.z`|
|`.r .g .b .a`|颜色语义|`c.r`, `c.g`, `c.b`|
|`.s .t .p .q`|纹理坐标语义|较少用|

**这四组写法是完全等价的别名（alias）**，比如：

```glsl
c.x  ==  c.r  ==  c.s   // 都是第一个分量
c.y  ==  c.g  ==  c.t   // 都是第二个分量
c.z  ==  c.b  ==  c.p   // 都是第三个分量
```

这也是为什么这段 hsb2rgb/rgb2hsb 代码里能看到 `c.bg`、`c.gb`、`c.r` 这种写法混用——**因为这个函数本质处理的是颜色数据，用 `.r/.g/.b` 在语义上更直观**（一看就知道是红绑绿蓝），而不是抽象的 `.x/.y/.z`，但底层访问的内存位置完全一样。

> [!addtion] ## 额外补充：GLSL 的"swizzle"语法
> 
> 你可能也注意到代码里出现了 `c.bg`、`c.gb` 这种**两个字母拼在一起**的写法，这叫 **swizzling（交换重组）**，可以任意组合、任意顺序、甚至重复取值：
> 
> ```glsl
> vec3 c = vec3(1.0, 2.0, 3.0);
> 
> c.x        // 1.0  （单个分量，取出来是float）
> c.xy       // (1.0, 2.0)  （取两个分量，组成vec2）
> c.bg       // (3.0, 2.0)  （等价于c.zy，顺序可以打乱）
> c.xxx      // (1.0, 1.0, 1.0)  （可以重复同一个分量三次）
> c.zyx      // (3.0, 2.0, 1.0)  （完全反转顺序）
> ```
> 
> 回到你的代码片段：
> 
> ```glsl
> vec4 p = mix(vec4(c.bg, K.wz), vec4(c.gb, K.xy), step(c.b, c.g));
> ```
> 
> - `c.bg` 是把 c 的 B、G 分量取出来，按"B在前G在后"的顺序组成一个新的二维数据
> - `c.gb` 则是反过来，"G在前B在后"
> 
> 这种 swizzle 语法正是 GLSL 写起来简洁但读起来一开始会觉得"密度很高"的原因——**一个看似随意的字母组合，背后是在重新排列/挑选向量的分量**，等你看顺眼了会发现这种写法其实很省事，不用像普通语言那样一个个分量单独赋值。
